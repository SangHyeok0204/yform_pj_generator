프론트엔드 개발 8대 철칙
1. 관심사의 철저한 분리 (Separation of Concerns)
원칙: 모든 컴포넌트는 "멍청하게(Dumb)" 유지한다.

세부 지침:

UI 컴포넌트: 로직을 갖지 않고 오직 Props를 통해 받은 데이터를 렌더링하고 이벤트를 전달만 한다.

비즈니스 로직: 데이터 가공, 상태 업데이트, API 호출 등은 Hooks나 Services 계층에서 처리한다.

내비게이션: 화면 이동 로직은 컴포넌트 내부가 아닌 Coordinator나 전용 라우터 핸들러에서 관리한다.

2. 도메인 중심의 수직 슬라이스 구조 (Vertical Slices)
원칙: 기술 스택(Components, Hooks)이 아닌 기능(Feature) 단위로 폴더를 구성한다.

세부 지침:

src/features/auth, src/features/dashboard와 같이 기능별로 독립된 폴더를 만든다.

각 슬라이스는 자신에게 필요한 UI, 로직, API 명세를 모두 포함하여 독립성을 보장한다.

이 구조를 통해 폴더 구조만 보고도 서비스의 성격을 파악할 수 있게 한다 (Screaming Architecture).

3. '신뢰할 수 없는 데이터'에 대한 엄격한 검증
원칙: 외부 데이터(API)는 반드시 런타임 검증을 거친 후 앱 내부로 진입시킨다.

세부 지침:

Zod를 사용하여 API 응답 스키마를 정의하고 검증한다.

앱 내부에서 사용하는 모델과 API 응답 모델을 분리하여, 서버 명세 변경이 앱 전체에 영향을 미치지 않도록 어댑터를 구현한다.

4. 렌더링 최적화 및 상태 관리 전략
원칙: 불필요한 네트워크 요청과 렌더링을 최소화한다.

세부 지침:

서버 상태 관리는 TanStack Query(React Query)를 사용하여 캐싱과 stale-while-revalidate 전략을 적용한다.

무분별한 전역 상태(Redux, Zustand) 사용을 지양하고, 최대한 지역 상태(Local State)로 해결한다.

불필요한 리렌더링 방지를 위해 memo, useMemo를 적재적소에 사용하되, 가독성을 해치지 않는 선을 지킨다.

5. 선언적 UI와 타입 안전성 (Type Safety)
원칙: 모든 코드는 명확한 타입을 가져야 하며, '어떻게'보다 '무엇을'에 집중한다.

세부 지침:

any 사용을 절대 금지하며, 인터페이스와 타입을 엄격히 정의한다.

UI 로직은 조건문 지옥을 피하기 위해 선언적(Declarative)으로 작성한다. (예: 복잡한 삼항 연산자 대신 매핑 오브젝트나 전용 컴포넌트 사용)

6. 성능 중심의 전송 전략
원칙: 사용자에게 가장 빠른 첫 화면 경험을 제공한다.

세부 지침:

정적 데이터는 SSG/ISR을, 개인화 데이터는 SSR 또는 PPR을 적절히 혼합한다.

거대한 라이브러리는 Lazy Loading과 Code Splitting을 통해 필요할 때만 불러온다.

이미지와 폰트 등 정적 리소스는 최적화된 포맷(WebP 등)과 레이지 로딩을 적용한다.

7. 테스트 가능성 (Testability)
원칙: 모든 비즈니스 로직은 UI 없이도 테스트 가능해야 한다.

세부 지침:

비즈니스 로직을 순수 함수(Pure Function)로 작성하여 유닛 테스트 작성을 용이하게 한다.

중요한 사용자 흐름은 Cypress나 Playwright를 통한 E2E 테스트 시나리오를 고려하여 마크업(data-testid 등)을 작성한다.

8. 문서화와 일관된 컨벤션
원칙: AI가 작성한 코드임을 알 수 없을 정도로 일관성을 유지한다.

세부 지침:

ESLint와 Prettier 규칙을 엄격히 준수한다.

새로운 기능을 만들 때는 반드시 해당 기능의 설계 의도와 사용법을 README.md 또는 JSDoc으로 남긴다.

클로드코드에게 주는 팁:

"위 철칙에 따라 코드를 작성해줘. 특히 Vertical Slices 구조를 유지하고, 모든 API 호출은 Zod로 검증하며, UI와 비즈니스 로직을 분리하는 Custom Hooks 패턴을 적극적으로 사용해."
